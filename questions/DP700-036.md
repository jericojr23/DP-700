#### Question

A bike-share operations team wants an eventstream transformation that calculates the total number of available bikes for each street every 5 seconds before loading the result to Eventhouse. Which Group by configuration matches the requirement?

#### Choices

- A. Sum `No_Bikes`, group by `Street`, use a tumbling window with a 5-second duration and 0-second offset
- B. Average `No_Bikes`, group by `Neighborhood`, use a hopping window with a 5-minute duration
- C. Count events, group by `Window_End_Time`, and omit a time window because Eventhouse adds one during ingestion
- D. Sum `No_Bikes`, do not group by any field, and query `Street` later from the raw `bikes` table

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Sum `No_Bikes`, group by `Street`, use a tumbling window with a 5-second duration and 0-second offset**

#### Explanation

The lab configures a Group by transformation named `GroupByStreet` with aggregate type Sum on `No_Bikes`, groups the aggregation by `Street`, and uses a tumbling time window with 5-second duration and 0-second offset. That configuration calculates the total number of bicycles in each street for each 5-second window before the output is stored.

#### Why the other choices are incorrect

- **A:** This matches the required metric, grouping field, window type, duration, and offset.
- **B:** Averaging by neighborhood with a hopping 5-minute window changes both the metric and the required street-level 5-second window.
- **C:** Counting events by `Window_End_Time` would not sum available bikes by street, and the window is defined in the Eventstream transformation rather than added automatically by Eventhouse ingestion.
- **D:** Omitting the group-by field produces an overall aggregate, not one result per street for the operations team.
