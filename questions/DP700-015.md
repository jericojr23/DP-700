### DP700-015: Refresh warehouse statistics after a major load

#### Question

A large Fabric Warehouse load significantly changed the row count and distribution of `CustomerKey` in `dbo.FactSales`. Investigation shows that the existing user-defined statistic `FactSales_CustomerKey` is stale. An SLA-critical join will run next, so the statistic must be refreshed immediately from all rows. Which command should you use?

#### Choices

- A. `UPDATE STATISTICS dbo.FactSales (FactSales_CustomerKey) WITH FULLSCAN`
- B. `DBCC SHOW_STATISTICS ('dbo.FactSales', 'FactSales_CustomerKey')`
- C. `VACUUM dbo.FactSales RETAIN 168 HOURS`
- D. `OPTIMIZE dbo.FactSales VORDER`

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. `UPDATE STATISTICS dbo.FactSales (FactSales_CustomerKey) WITH FULLSCAN`**

#### Explanation

Accurate statistics help the Warehouse query optimizer estimate row counts and choose efficient plans. Microsoft recommends updating column statistics after data changes that materially alter row count or distribution; `WITH FULLSCAN` refreshes the named user-defined statistic from all table rows before the critical query.

#### Why the other choices are incorrect

- **A:** This explicitly refreshes the identified statistic with the complete current data distribution.
- **B:** `DBCC SHOW_STATISTICS` displays the statistic and histogram but does not update either one.
- **C:** `VACUUM` is Delta-file cleanup and does not refresh Warehouse optimizer statistics.
- **D:** V-Order changes Parquet file layout for reads; it does not update the named histogram statistic.

</details>
