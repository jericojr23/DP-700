### DP700-030: Route event subsets by content

#### Question

An eventstream receives IoT sensor events from a factory. High-temperature events must be routed to Fabric Activator for alerting, while hourly aggregate events must be routed to a KQL database for analysis. The team wants the routing to be based on event content without changing the original stream for other consumers. Which eventstream destination pattern should it use?

#### Choices

- A. Create derived streams from the original stream and route each derived stream to the appropriate destination
- B. Send every event only to a Lakehouse destination and create alerts from the Delta table later
- C. Replace the eventstream with a Dataflow Gen2 item that filters the IoT events
- D. Create a custom endpoint and require downstream systems to split the event content

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Create derived streams from the original stream and route each derived stream to the appropriate destination**

#### Explanation

Derived streams are transformed versions of the original data stream and support content-based routing. The unit gives the pattern of filtering IoT sensor data so high-temperature events can go to Fabric Activator while hourly averages go to a KQL database.

#### Why the other choices are incorrect

- **A:** Derived streams preserve the content-based routing requirement and allow different destinations for different processed subsets.
- **B:** Sending everything only to a Lakehouse delays the immediate alerting requirement and does not use the eventstream routing pattern described.
- **C:** Dataflow Gen2 is not the eventstream destination pattern for routing real-time stream subsets to Activator and KQL.
- **D:** A custom endpoint can send events to an external application, but it shifts the routing work out of Fabric and does not satisfy the requested in-eventstream content-based routing.
