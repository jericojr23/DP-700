#### Question

A Real-Time Dashboard tile initially displays a KQL query result as a table with `Neighbourhood`, `No_Bikes`, and `No_Empty_Docks`. The operations team wants the same result to be easier to compare visually by neighborhood. What should the dashboard editor do?

#### Choices

- A. Edit the tile, select a bar chart visual, and map the neighborhood and bike-count fields in the visual formatting pane
- B. Create a Fabric warehouse and move the query result into a relational table before displaying it
- C. Replace the dashboard with an Eventstream transformation because tables cannot be visualized
- D. Add a text tile that describes the values and leave the data tile as an unformatted table

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Edit the tile, select a bar chart visual, and map the neighborhood and bike-count fields in the visual formatting pane**

#### Explanation

After creating a Real-Time Dashboard tile, the editor can change how the query results are represented by choosing a visual such as a chart or map and configuring the visual fields. For bike availability by neighborhood, a bar chart with neighborhood on the category axis and bike-count fields as values matches the stated comparison requirement.

#### Why the other choices are incorrect

- **A:** This keeps the KQL-backed dashboard tile and changes only the visual representation to match the comparison need.
- **B:** A warehouse is not required to visualize a KQL query result in a Real-Time Dashboard tile.
- **C:** Eventstream transformations process events in motion, but the scenario already has a tile query result that can be formatted as a visual.
- **D:** A text tile can provide additional information, but it does not make the numeric query results easier to compare visually.
