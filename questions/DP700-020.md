#### Question

A Fabric notebook repeatedly runs large aggregations and joins against Delta tables in OneLake. The code already uses supported Spark SQL and DataFrame operations, and the team wants acceleration without rewriting the transformations. The setting should apply to every new notebook session that uses the team's environment. What should you do?

#### Choices

- A. Enable the native execution engine in the environment's Acceleration settings, then save and publish the environment
- B. Convert the Delta tables to JSON files and keep the existing notebook code
- C. Disable adaptive query execution so every query uses a fixed physical plan
- D. Replace the notebook with a Dataflow Gen2 item to use Spark native execution automatically

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Enable the native execution engine in the environment's Acceleration settings, then save and publish the environment**

#### Explanation

The Fabric native execution engine can accelerate supported Spark SQL and DataFrame workloads without code changes by offloading supported operators to a vectorized native execution path. Enabling it at the environment level and publishing the change lets subsequent notebooks and jobs attached to that environment inherit the setting.

#### Why the other choices are incorrect

- **A:** This applies the acceleration setting at the reusable environment level and matches the supported Delta aggregation and join scenario.
- **B:** JSON queries are not the target acceleration path for the native execution engine, and converting from Delta would discard the stated table format advantage.
- **C:** The native engine preserves Fabric Spark optimizations such as adaptive query execution; disabling it is not the documented enablement path.
- **D:** Dataflow Gen2 is not the mechanism for enabling the Spark native execution engine for notebook sessions.
