### DP700-016: Query one lakehouse only through T-SQL

#### Question

A Fabric workspace contains separate lakehouses for raw, refined, and curated data. A Microsoft Entra group of reporting analysts must run T-SQL queries against every Delta table in the curated lakehouse. The analysts must not receive access to the other workspace items or read the curated lakehouse through Apache Spark or OneLake APIs. Which access configuration should you use?

#### Choices

- A. Assign the group the workspace Viewer role
- B. Share the curated lakehouse with the group and grant **Read all with SQL analytics endpoint**
- C. Share the curated lakehouse with the group and grant **Read all with Apache Spark**
- D. Share only the curated lakehouse's default semantic model with the group and grant Build permission

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**B. Share the curated lakehouse with the group and grant Read all with SQL analytics endpoint**

#### Explanation

Sharing the curated lakehouse provides item-level access without making the analysts members of the workspace. The additional **Read all with SQL analytics endpoint** permission allows them to query all of its Delta tables through T-SQL without granting access through Apache Spark or OneLake APIs.

#### Why the other choices are incorrect

- **A:** Viewer applies to every item in the workspace, so it exposes the raw and refined lakehouses as well as the curated lakehouse.
- **B:** This combines item-level sharing with the required SQL-only data-access path and avoids workspace-wide access.
- **C:** Read all with Apache Spark permits access through Spark and OneLake APIs, which the scenario explicitly prohibits.
- **D:** Build permission supports creating content from a semantic model; it does not grant the required T-SQL access to every Delta table through the lakehouse's SQL analytics endpoint.
