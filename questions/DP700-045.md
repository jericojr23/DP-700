#### Question

A developer opens the SQL analytics endpoint for a Fabric lakehouse and runs `UPDATE dbo.Sales SET Status = 'Closed' WHERE OrderId = 12045;` against a Delta table. The statement fails. The team needs to understand why before choosing the correct write path. What is the best explanation?

#### Choices

- A. The SQL analytics endpoint is read-only for lakehouse Delta tables; use Apache Spark in the lakehouse or a Fabric Warehouse write surface for data modifications.
- B. The statement failed only because the `WHERE` clause filters on a numeric column instead of a string column.
- C. `UPDATE` is unsupported everywhere in Microsoft Fabric, including Warehouse tables.
- D. The statement would work if the user first created a dashboard tile over `dbo.Sales`.

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. The SQL analytics endpoint is read-only for lakehouse Delta tables; use Apache Spark in the lakehouse or a Fabric Warehouse write surface for data modifications.**

#### Explanation

The lakehouse SQL analytics endpoint provides a read-only T-SQL query surface over Delta tables. Microsoft documentation states that insert, update, and delete operations are not supported through the lakehouse SQL analytics endpoint; data modifications must use an appropriate write-capable surface such as Spark in the lakehouse or Warehouse when the data is modeled there.

#### Why the other choices are incorrect

- **A:** This identifies the endpoint limitation and points the team to a write-capable Fabric surface.
- **B:** Filtering by a numeric key is valid SQL syntax; the failure is due to the endpoint's read-only boundary, not the predicate type.
- **C:** Fabric Warehouse supports write operations such as insert, update, and delete; the limitation applies to the lakehouse SQL analytics endpoint.
- **D:** A dashboard tile can visualize query results but cannot change the T-SQL write capabilities of the endpoint.
