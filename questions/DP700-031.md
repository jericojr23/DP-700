### DP700-031: Use multiple destinations after a transformation

| Field | Value |
|---|---|
| Domain | Ingest and transform data |
| Topic | Eventstream destinations after transformations |
| Difficulty | Advanced |

| Source | [Eventstream sources and destinations](https://learn.microsoft.com/en-us/training/modules/explore-event-streams-microsoft-fabric/3-setup-eventstreams) |

| Status | Verified |

#### Question

A city mobility team uses an eventstream to group bicycle availability events by street. The transformed output must be inserted into an Eventhouse table for KQL analysis and must also drive an Activator rule that checks whether stations have available bikes. Both consumers need the same transformed output, and neither path should block or replace the other. What should the team configure?

#### Choices

- A. Attach both an Eventhouse destination and a Fabric Activator destination after the transformation
- B. Create only an Eventhouse destination because an eventstream can have only one active destination
- C. Route the stream to a custom endpoint and then manually reload the data into Eventhouse and Activator
- D. Store the events in a Lakehouse first because Activator cannot consume eventstream output

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Attach both an Eventhouse destination and a Fabric Activator destination after the transformation**

#### Explanation

Eventstream destinations can be configured after a data source is connected or after optional transformations are applied. The unit states that multiple destinations can be attached to the same eventstream at the same time without colliding, and its example routes transformed `GroupByStreet` output to both Activator and Eventhouse.

#### Why the other choices are incorrect

- **A:** This keeps the transformed output inside the eventstream and sends it to both required destinations independently.
- **B:** The unit explicitly supports multiple destinations in an eventstream, so limiting the flow to one destination is unnecessary.
- **C:** A custom endpoint is useful for external systems, but it adds avoidable manual routing and does not use the built-in Eventhouse and Activator destinations.
- **D:** Eventstreams can connect directly to Fabric Activator as a destination, so a Lakehouse staging step is not required for the alerting path.

</details>
