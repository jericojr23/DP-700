#### Question

A team develops a SQL database in Fabric. It must track table, view, and stored-procedure definitions in Azure DevOps, review changes as code, and apply validated differential schema changes back to the live database. The team does not need to version table data. Which approach should it use?

#### Choices

- A. Connect the workspace to a Git repository and use the SQL database source-control integration
- B. Create a OneLake shortcut to the SQL database
- C. Schedule a Copy job to copy every database table to a lakehouse
- D. Use the Monitoring hub to retain the database definitions

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Connect the workspace to a Git repository and use the SQL database source-control integration**

#### Explanation

Fabric SQL database Git integration converts the live database object definitions into code when committing and validates repository code before applying differential changes during an update. This matches the requirement to version schema rather than data.

#### Why the other choices are incorrect

- **A:** Git integration provides the required schema history, code review workflow, validation, and differential update behavior.
- **B:** A shortcut exposes existing data without copying it; it is not a database-object version-control mechanism.
- **C:** A Copy job moves data and would duplicate table contents, but it does not version SQL object definitions.
- **D:** Monitoring hub records workload runs; it is not a source repository for database schema.

</details>
