#### Question

A Real-Time Dashboard has a parameter named `_eventType`. The parameter appears in the dashboard parameter list, but selecting a value does not change one of the visuals. The visual's KQL query does not reference `_eventType`. What should you do?

#### Choices

- A. Add `_eventType` to the visual's KQL query, such as in a `where` filter, and save the visual
- B. Change the parameter label so it matches the tile title exactly
- C. Move the parameter card to the top of the Parameters pane
- D. Replace the KQL database data source with a semantic model

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Add `_eventType` to the visual's KQL query, such as in a `where` filter, and save the visual**

#### Explanation

Real-Time Dashboard parameters are managed at the dashboard level, but they affect a visual only when the visual's query uses the parameter variable. Microsoft documentation states that if a query does not use the parameter, the filter remains inactive. Adding the variable to a `where` condition makes the selected parameter value filter that visual.

#### Why the other choices are incorrect

- **A:** This connects the dashboard parameter to the visual's query logic, so the selected filter value can affect the result.
- **B:** The display label does not activate filtering; the query must use the parameter variable.
- **C:** Reordering parameter cards changes manageability, not whether a tile query uses the parameter.
- **D:** The issue is missing parameter usage in the KQL query, not the type of data source.
