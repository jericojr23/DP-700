#### Question

Frequent ingestion has left a read-heavy lakehouse Delta table with many small files. Reports repeatedly filter on `OrderDate`, and the table is scanned by multiple Fabric engines. Which Spark SQL maintenance action best addresses all three conditions in one operation?

#### Choices

- A. `OPTIMIZE sales ZORDER BY (OrderDate) VORDER`
- B. `VACUUM sales RETAIN 168 HOURS`
- C. `UPDATE STATISTICS sales (OrderDate) WITH FULLSCAN`
- D. `DROP TABLE sales` followed by a full reload

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. `OPTIMIZE sales ZORDER BY (OrderDate) VORDER`**

#### Explanation

`OPTIMIZE` bin-compacts small files, `ZORDER BY (OrderDate)` colocates values to improve data skipping for the recurring selective filter, and `VORDER` optimizes the Parquet layout for broad read efficiency across Fabric engines. The combined command targets every stated performance condition.

#### Why the other choices are incorrect

- **A:** It combines compaction, filter-column clustering, and cross-engine file-layout optimization in one supported Spark SQL operation.
- **B:** `VACUUM` removes old unreferenced files; it does not compact current files or cluster current data for the filter.
- **C:** Statistics can improve SQL plan estimates, but this action does not fix the Delta small-file layout or apply Z-Order and V-Order.
- **D:** A destructive full reload is unnecessary and does not ensure the requested optimized physical layout.
