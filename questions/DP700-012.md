#### Question

A newly authored Fabric pipeline has six activities, and one transformation is failing. You want to isolate the activity and validate its parameters without executing unrelated downstream work. What should you do?

#### Choices

- A. Deactivate the unrelated activities, save and run the pipeline, and inspect the result in Monitoring hub
- B. Start a separate authoring-time Debug run without saving the pipeline
- C. Delete the downstream activities and recreate them after testing
- D. Promote the pipeline to production and review the deployment comparison

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Deactivate the unrelated activities, save and run the pipeline, and inspect the result in Monitoring hub**

#### Explanation

Fabric Data Factory has no separate authoring-time Debug mode. Its supported debugging workflow is to deactivate activities to isolate logic, save and run the pipeline, and then review the execution results in Monitoring hub.

#### Why the other choices are incorrect

- **A:** Deactivation skips the isolated activities without destroying their definitions and supports incremental testing.
- **B:** A separate unsaved Debug execution is an Azure Data Factory pattern, not the Fabric pipeline debugging model described in the documentation.
- **C:** Deleting working activities is unnecessary and risks losing their configuration.
- **D:** Deployment is a lifecycle-management action and should not be used to diagnose unvalidated logic in production.

</details>
