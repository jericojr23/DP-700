#### Question

An operations dashboard monitors several Eventhouse-backed visuals. The team wants visuals to update when new data is ingested, but it wants to avoid constantly rerunning heavy tile queries during quiet periods. Unsupported visuals should still refresh on a controlled interval. Which configuration should the dashboard editor use?

#### Choices

- A. Enable Live refresh in Refresh settings, set a refresh rate limit, and set a fallback refresh interval
- B. Configure Manual update only and ask viewers to select Update whenever they need current data
- C. Schedule a Data Factory pipeline to reload every KQL query behind the dashboard
- D. Create an Activator rule for each visual and use the rule state as the dashboard refresh trigger

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Enable Live refresh in Refresh settings, set a refresh rate limit, and set a fallback refresh interval**

#### Explanation

Live refresh updates Real-Time Dashboard visuals when new data is ingested into the underlying source. It uses lightweight ingestion detection for supported visuals and avoids rerunning heavy visual queries when no new data is detected. Editors can configure a refresh rate limit and a fallback refresh interval for visuals or data sources that do not support ingestion detection.

#### Why the other choices are incorrect

- **A:** This matches the requirements for ingestion-aware updates, throttling, and controlled fallback behavior.
- **B:** Manual update only prevents automatic refresh and does not satisfy the need to update visuals when new data arrives.
- **C:** A Data Factory pipeline is not the Real-Time Dashboard refresh mechanism and would add unnecessary orchestration outside the dashboard.
- **D:** Activator triggers alerts or automated actions from rules; it is not the documented refresh engine for dashboard visuals.
