#### Question

A support team has application telemetry in a KQL database. The team needs a Fabric dashboard page with a tile that shows the count of recent errors by severity, and the tile must be based on a KQL query against that database. What should the team configure?

#### Choices

- A. Add the KQL database as a Real-Time Dashboard data source, then create a tile with a KQL query and visual formatting
- B. Create a Dataflow Gen2 item and pin its refresh history to the dashboard
- C. Create a Fabric Activator rule and use the activation history as the visual query
- D. Add a workspace role assignment and wait for the dashboard to infer visuals from all workspace items

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. Add the KQL database as a Real-Time Dashboard data source, then create a tile with a KQL query and visual formatting**

#### Explanation

Real-Time Dashboards are collections of tiles. Each tile has an underlying query and a visual representation, and the dashboard can connect to a KQL database data source. A tile can be authored manually with a KQL query or added from a KQL queryset, then formatted as the required visual.

#### Why the other choices are incorrect

- **A:** This provides both required parts: a connected KQL database data source and a tile backed by a KQL query.
- **B:** Dataflow Gen2 refresh history does not create a KQL-backed Real-Time Dashboard visual over telemetry data.
- **C:** Activator is for rule-based actions and alerts; its activation history is not the dashboard tile authoring path for querying telemetry.
- **D:** Workspace roles control permissions and do not cause dashboards to automatically infer KQL visuals from workspace items.
