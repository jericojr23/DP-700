#### Question

An eventstream is continuously loading bicycle observations into a KQL table named `bikes` in an Eventhouse. After publishing the destination, you need to verify that records were ingested during the last 24 hours. Which KQL query should you run?

#### Choices

- A. `bikes | where ingestion_time() between (now(-1d) .. now())`
- B. `bikes | summarize count() by Street | where ingestion_time() > now()`
- C. `bikes | take 100 | where Window_End_Time between (now(-1d) .. now())`
- D. `bikes | where CreatedDate == today() | project ingestion_time()`

<details>
<summary>Answer and explanation</summary>

#### Correct answer

**A. `bikes | where ingestion_time() between (now(-1d) .. now())`**

#### Explanation

The lab uses the KQL database table menu to generate a query for records ingested in the last 24 hours. The query reads from `bikes` and filters by `ingestion_time()` between `now(-1d)` and `now()`, which checks the ingestion timestamp rather than a business field from the event payload.

#### Why the other choices are incorrect

- **A:** This filters the Eventhouse table by the ingestion-time range requested for the last 24 hours.
- **B:** This groups by street before applying an invalid future-looking ingestion-time filter and would not return the requested recently ingested records.
- **C:** `Window_End_Time` belongs to the transformed grouped output in the lab, not the raw `bikes` table used to verify captured records.
- **D:** The lab does not rely on a `CreatedDate` payload field or a `today()` equality comparison to validate ingestion recency.
